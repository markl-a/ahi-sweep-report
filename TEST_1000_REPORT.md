# 957 人 Held-Out 測試報告

> **用 957 個模型從未見過的受試者，測試 tcn_bilstm_hybrid 的真實表現。**

---

## 1. 測試設計

### 為什麼是 957 人而不是 1000 人？

我們從 SHHS 資料庫中挑選了 1000 個**完全沒參與訓練或驗證的受試者**（held-out test set）。其中 43 人缺少預處理後的訊號檔案（npz），最終有效測試人數為 **957 人**。

### 跟之前的差別

| | Sweep 評估 | Per-Subject 分析 | **本次 Held-Out Test** |
|---|---|---|---|
| 測試人數 | 20 | 156 | **957** |
| 這些人參與過訓練嗎？ | 否（val set） | 否（val set） | **完全沒有（held-out）** |
| 這些人被用來選模型嗎？ | 是（early stop） | 是 | **否** |
| AHI r 可信度 | 低（N=20） | 中（N=156） | **高（N=957）** |
| 測試嚴格度 | ⭐ | ⭐⭐ | **⭐⭐⭐** |

> **這是最嚴格的測試方式。** 模型在訓練時完全沒見過這 957 個人的任何資料。就像讓一個學生做一份他從沒看過的考卷。

### 模型與訓練

- **架構**：tcn_bilstm_hybrid（trial 721 config）
- **參數**：nf=128, nl=12, k=7, lr=0.003, loss=bce
- **訓練資料**：624 個 SHHS 受試者（跟 sweep 相同的 80/20 split 的 train 部分）
- **訓練量**：30 epochs × 1000 chunks（精簡訓練，完整訓練的 1/20）
- **運算設備**：NVIDIA RTX 5070 (CUDA)
- **訓練時間**：530 秒（約 9 分鐘）

## 2. 全部指標一覽

### 2.1 逐樣本指標（per-sample，每 0.1 秒為一單位）

> 這些指標把整晚的呼吸訊號切成每 0.1 秒一段，看模型對每一段的判斷是否正確。

| 指標 | 數值 | 白話解釋 |
|---|---|---|
| **AUC-ROC** | **0.886** | 模型區分「有事件」和「沒事件」的整體能力。0.5 = 亂猜，1.0 = 完美。0.886 = **很好** |
| **Accuracy** | 0.883 | 每 0.1 秒的判斷中，88.3% 是正確的 |
| **F1** | **0.616** | 精確度和召回率的綜合分數 |
| **Recall** | 0.562 | 真實的呼吸事件中，有 **56.2%** 被模型抓到了 |
| **Precision** | 0.682 | 模型預測的呼吸事件中，有 **68.2%** 是真的 |
| **Specificity** | 0.948 | 沒有呼吸事件的時段，**94.8%** 被模型正確判為「正常」 |

> **解讀**：
> - **AUC-ROC = 0.886** 是最重要的指標，因為它不受偵測門檻影響。0.886 代表模型有很強的區分能力。
> - **Recall = 56.2%** 表示模型漏掉了約 44% 的真實事件。這是改善空間最大的地方。
> - **Precision = 68.2%** 表示模型報的警報中約 32% 是虛驚。
> - **Specificity = 94.8%** 表示模型很少在沒事的時候亂報警（假警報率低）。

### 2.2 逐事件指標（per-event，IoU ≥ 50%）

> 這些指標看的是「事件級別」：模型預測的每一個「呼吸停止事件」跟真實標註的事件有沒有 ≥50% 的重疊（IoU matching）。

| 指標 | 數值 | 白話解釋 |
|---|---|---|
| **Event F1** | **0.535** | 事件級別的綜合準確度 |
| **Event Recall** | 0.544 | 真實事件中有 54.4% 被找到（與真實標註重疊 ≥50%） |
| **Event Precision** | 0.527 | 模型預測的事件中 52.7% 跟真實事件匹配 |

> **解讀**：
> - Event F1 = 0.535 比 sample F1 = 0.616 低，因為 event-level 要求更嚴格（不只要判對 0/1，還要定位邊界）
> - Event Recall ≈ Event Precision，代表模型不偏向「多報」或「少報」，算是平衡的

### 2.3 AHI 預測相關性

| 指標 | 數值 | 說明 |
|---|---|---|
| **AHI Pearson r** | **0.646** | 957 人上的 AHI 預測相關性 |
| p 值 | 4.05 × 10⁻¹¹⁴ | 極度顯著（不是隨機的） |

> **解讀**：
> - r = 0.646 代表模型預測的 RDI 跟真實 AHI 有中等偏高的相關性
> - 之前 sweep 中 N=20 時算出 r=0.911，**現在 N=957 時降為 0.646**，證實了之前的 caveat（小樣本 r 值不可靠）
> - **這是未校正的 raw RDI vs AHI。** Phase B v2 在校正後達到 r=0.840。如果對本模型也做校正，r 預期會提升
> - **這是 1/20 訓練量的結果。** 完整訓練預期會顯著提升

### 2.4 嚴重度分級準確度

| 指標 | 數值 |
|---|---|
| **四級分類準確率** | **32.5%**（311/957）|

> **看起來很低，但原因不是模型差 — 是 AHI 定義問題。** 詳見 §3。

### 3.1 每個嚴重度的表現

| 嚴重度 | 人數 | Event F1 | AHI r | 分級準確率 |
|---|---|---|---|---|
| **severe** (≥30) | 161 | **0.657** | 0.438 | **83.2%** ✅ |
| moderate (15-30) | 243 | 0.545 | 0.223 | 28.0% |
| mild (5-15) | 337 | 0.451 | 0.213 | 25.8% |
| normal (<5) | 216 | 0.405 | 0.280 | 10.2% |

> **重症偵測最準（83.2%），輕症最不準（10.2%）。**

### 3.2 嚴重度混淆矩陣

> 這張表讀法：每一列是「真實嚴重度」，每一欄是「模型預測的嚴重度」。對角線（粗體）是正確的。

|  | 預測 normal | 預測 mild | 預測 moderate | 預測 severe |
|---|---|---|---|---|
| **真實 normal** (216) | **22** | 97 | 70 | 27 |
| **真實 mild** (337) | 3 | **87** | 146 | 101 |
| **真實 moderate** (243) | 5 | 21 | **68** | 149 |
| **真實 severe** (161) | 2 | 5 | 20 | **134** |

### 3.3 混淆矩陣解讀

**模型有一個明顯的偏向：把人往更嚴重的方向判。**

具體數字：
- 216 個 normal 的人，只有 22 個被正確判為 normal，**97 個被誤判為 mild，70 個被誤判為 moderate**
- 337 個 mild 的人，**146 個被誤判為 moderate，101 個被誤判為 severe**

> **為什麼會這樣？**
>
> 這個問題在 Per-Subject Analysis 中已經發現：**AHI 定義跟事件標註不匹配。**
>
> - AHI（`ahi_a0h3_official`）只算**伴隨 ≥3% 血氧下降**的事件
> - 我們的模型學的是**所有呼吸事件**（包含不伴隨血氧下降的）
> - 所以模型的 predicted RDI 總是比 manifest AHI **高**
> - 一個 manifest AHI = 3（normal）的人，可能有 RDI = 15（moderate 級別的事件數量）
>
> **模型做對了**（它確實偵測到了呼吸事件），但**我們的評分標準跟它學的東西不一致**。
>
> 就像一個學生學了「所有水果都算水果」，但考試只承認「蘋果和橘子是水果」。學生答「番茄是水果」被算錯 — 但學生的知識其實是對的。

## 5. AUC-ROC 的意義

AUC-ROC = 0.886 是本次測試中**最可靠也最有價值的指標**。

> **為什麼 AUC-ROC 比 F1 更可靠？**
>
> F1 受到偵測門檻（threshold = 0.4）的影響。如果我們把門檻從 0.4 調到 0.3 或 0.5，F1 會變。但 AUC-ROC **不受門檻影響** — 它衡量的是「模型給真實事件的分數是否高於非事件」的整體能力。
>
> 0.886 代表：如果隨機挑一個「有事件的 0.1 秒」和一個「沒事件的 0.1 秒」，模型有 **88.6% 的機率**給有事件的那段更高的分數。

### 門檻調整的空間

目前使用 threshold = 0.4。如果調整：
- 降低 threshold（例如 0.3）→ Recall ↑, Precision ↓, 更多事件被抓到但也更多假警報
- 提高 threshold（例如 0.5）→ Recall ↓, Precision ↑, 更少假警報但也漏更多事件

**AUC-ROC = 0.886 保證了不管怎麼調門檻，都有很好的 trade-off 空間。**

## 6. 改善方向

根據本次 957 人測試的結果，以下是按優先級排序的改善方向：

### P0：統一 AHI 定義（影響最大）

**問題**：模型學的是「所有呼吸事件」，但 AHI 只算部分事件（a0h3 規則）。

**解法**：
- (a) 重新計算 AHI，改用 RDI（所有事件）作為 ground truth → r 預估提升 +0.10~0.20
- (b) 修改事件標註，只保留伴隨 ≥3% 血氧下降的事件 → 更精準匹配 AHI
- (c) 在後處理加入 AHI 校正（多項式回歸）→ Phase B v2 已驗證有效

### P1：增加訓練量

| 動作 | 預估效果 | 成本 |
|---|---|---|
| 60 epochs（×2） | F1 +0.05~0.10 | +9 min |
| 10000 chunks/epoch（×10） | F1 +0.03~0.05 | +90 min |
| 3-ensemble（×3） | F1 +0.02~0.03 | ×3 時間 |
| **全部加起來** | **F1 +0.10~0.18** | ~4 小時 |

### P2：優化偵測門檻

目前 threshold 固定 0.4。可以在 val set 上做 threshold sweep（0.2~0.6），找到 F1 最高的門檻。

### P3：在輔大醫院資料上驗證

確認在台灣在地族群上的泛化能力。

## 9. 技術備註

### 為什麼重新訓練而非載入模型？

**因為之前的訓練都沒有存模型的 weights（權重）到磁碟。** 這是設計疏失。

`trial_runner.py` 和 `per_subject_analysis.py` 只輸出 result JSON、heatmap PNG 和 CSV，但沒有加 `torch.save(model.state_dict(), 'checkpoint.pth')` 這一行。所以沒有 `.pth` 檔案可以載入，必須重新訓練。

重新訓練用的是**完全相同的 config、seed、訓練資料**，所以模型幾乎一樣（CUDA 浮點計算可能有微小差異，不影響結論）。
